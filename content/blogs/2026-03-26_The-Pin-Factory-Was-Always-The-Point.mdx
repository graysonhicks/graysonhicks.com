## A Room Full of Half-Jobs

In the late 1700s, Adam Smith visited a pin factory. Just a few workers, some wire, and a repetitive process turning out something as trivial as pins.

What caught his attention wasn't the product, it was how the work had been divided.

No one in the room was actually making a pin.

![A pin factory, circa late 1700s](/images/blog/pin-factory.png)

One man pulled wire. Another straightened it. Another cut it. Someone else sharpened the ends, someone formed the heads, another attached them, and others finished and packed. Isolated, each contribution looked almost useless, just a fragment of a thing. Together, they were producing tens of thousands of pins a day.

A man working alone might make twenty, not because he's lazy or incapable, but because he's constantly switching between steps, constantly reorienting himself.

The difference wasn't talent. It was that no one had to carry the entire process at once.

---

## Where the Intelligence Actually Lives

The division wasn't arbitrary. Each step was small, but more importantly, it was stable and easy to evaluate. You could look at the output of a single step and know whether it was right without having to understand everything that came before or after.

Over time, more of the intelligence starts living not in the worker, but in how the steps connect.

That principle shows up everywhere systems get built, from assembly lines to software pipelines. And it turns out it maps almost directly onto one of the harder problems in AI right now: getting multiple AI agents to work together reliably. You don't really appreciate how much the division matters until you try to build a multi-agent system without it.

---

## One Agent Doing Everything

Most people building multi-agent systems don't start that way.

You take a real problem and hand the whole thing to a single agent: analyze this dataset, write the report, make a decision, format the output.

For a while, it works. It even feels like you've found a shortcut, skipping all the intermediate structure and going straight from input to answer.

We had a system like this early on. One agent doing everything. It looked great in demos. Then we tried to make the output consistent across slightly different inputs.

That's where things started to drift.

Not in obvious ways:

* The format would shift a little
* The emphasis would move
* Sometimes it would skip a step it had handled correctly the run before

Nothing was clearly broken, which made it harder to diagnose.

So we did what everyone does. Tightened the prompt. Added rules. Then added rules for when those rules didn't apply. It would improve for a bit, then something else would slip.

At some point it stopped feeling like engineering and started feeling more like negotiation.

The issue wasn't that the model couldn't reason. It was that we were overloading it. These systems don't carry a stable internal state; they rebuild it from context every time, and that reconstruction depends on how the input is shaped, what's nearby, and what gets emphasized. Every additional responsibility is another thing the model has to reorient itself around mid-stream, another chance for something to slip.

---

## One Transformation at a Time

Instead of asking for the whole outcome, you ask for one transformation at a time.

Say you're building a system to process incident reports. Instead of one agent that reads, categorizes, extracts, and summarizes, you split it:

* Agent 1 extracts entities and metrics from the raw text
* Agent 2 normalizes them into a structured schema
* Agent 3 flags anomalies against historical baselines
* Agent 4 writes the summary, working only from validated fields

Each step is small enough to validate on its own. If extraction drifts, you catch it before it poisons the summary. If the schema changes, you update one agent instead of rewriting a monolithic prompt.

*(If you're wondering what I use to bring pipelines like this to life, it's [Mastra](https://mastra.ai).)*

By the time you get to the part that actually requires judgment, it's working on something that's already been shaped into a predictable form.

![The Pin Factory vs. A Multi-Agent System](/images/blog/pin-factory-multi-agent.png)

Once you start chaining steps like these together, the system stops behaving like a single pass and starts behaving like a pipeline. You can route based on outputs, reject results that don't meet a threshold, loop steps until they pass, or branch the process entirely. That only works if the underlying steps are already separated. Otherwise you're just layering more responsibility onto the same overloaded process.

---

## Knowing When to Split

The hard part isn't the separation itself. It's knowing when a single agent is carrying too much, and when to hand part of the job to a new one.

There's no formula for this. It's a judgment call, the same kind a factory manager makes when they notice one station is bottlenecking the line. You watch for the signs:

* Output quality getting inconsistent
* A prompt growing longer and more conditional
* One part of the task interfering with another

Those are the moments where a new agent earns its place.

The temptation is to split too early or too late. Too early, and you're managing coordination overhead for something a single agent handles fine. Too late, and you've already spent weeks patching a prompt that needed to be two prompts.

The best multi-agent systems we've built didn't start with a clean architecture diagram. They started with one agent, and we paid attention to where it struggled. Each new agent was a response to a specific failure, not a guess about where complexity might show up later.

---

## Building Systems You Can See Into

When everything happens inside one agent, a failure could be anywhere. Once the steps are separated, the boundaries give you something to look at. You can log what comes out of each step, test one part without touching the others, and improve a single piece while watching how it affects the rest of the system.

Tools narrow in the same way. Each step only has access to what it actually needs: retrieval handles search, calculation handles computation, formatting enforces structure. Less surface area, fewer mistakes.

---

## The Model Worth Carrying

The instinct, when things start to break, is to make the agent better. Add more instructions. Cover more cases. Make it "smarter." That's exactly what the pin factory worked against. The gains didn't come from improving the worker. They came from reducing what each worker had to do, and making the connections between them precise.

The moment you find yourself writing a long prompt that tries to do multiple things at once — enforce structure, reason, format, handle edge cases — you're fighting the same battle Smith's lone pin-maker was fighting.

The real design skill in agent systems isn't making a powerful agent. It's noticing the moment a task stops being one job and becomes several. The smaller-step systems don't feel as impressive. They just keep working.
