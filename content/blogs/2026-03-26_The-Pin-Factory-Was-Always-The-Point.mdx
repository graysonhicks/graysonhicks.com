## A Room Full of Half-Jobs

In the late 1700s, Adam Smith visited a pin factory. Just a few workers, some wire, and a repetitive process turning out something as trivial as pins.

What caught his attention wasn't the product, it was how the work had been divided.

No one in the room was actually making a pin.

![A pin factory, circa late 1700s](/images/blog/pin-factory.png)

One man pulled wire. Another straightened it. Another cut it. Someone else sharpened the ends, someone formed the heads, another attached them, and others finished and packed. Smith noted that a man working alone might make twenty pins in a day — maybe not even one. Together, those ten workers made forty-eight thousand.

Before factories like this existed, an entire village might possess a single pin. Not because pins were mysterious or the materials were scarce, but because making one from scratch, end to end, was genuinely hard. The factory didn't invent new skills. It just stopped asking one person to hold all of them at once.

The difference wasn't talent. It was that no one had to carry the entire process at once.

We might be at a similar inflection point right now with AI agents. The technology isn't new — the models can reason, extract, summarize, decide. But most of us are still in the village era, asking one agent to do everything from scratch every time.

## One Agent Doing Everything

At a previous company in the manufacturing financing space, we built an underwriting agent for banks. The job was straightforward on paper: ingest emails and their attachments — PDFs, Excel files, scanned documents — extract the relevant data, normalize it, analyze it against the bank's criteria, and propose a decision. Approve, deny, or flag for review.

We started with one agent doing all of it. It worked well enough in demos. Then we tried to make the output consistent across real submissions — different attachment formats, different email structures, slightly different ways of saying the same thing.

That's where things started to drift.

Not in obvious ways:

* Extraction would miss a field it caught last time because the PDF layout shifted
* The normalization would be inconsistent — the same data point formatted differently depending on what else was in the email
* Sometimes the analysis would bleed into the extraction, and the agent would start making judgments before it had finished collecting facts

Nothing was clearly broken, which made it harder to diagnose.

So we did what everyone does. Tightened the prompt. Added rules. Then added rules for when those rules didn't apply. It would improve for a bit, then something else would slip.

At some point it stopped feeling like engineering and started feeling more like negotiation. We were spending more time managing the agent's behavior than improving the actual underwriting logic.

---

## One Transformation at a Time

The fix wasn't making the agent smarter. It was making each agent do less.

Instead of one agent ingesting an email and producing an underwriting decision, we split it into steps that each did one transformation:

* An extraction agent that pulled data from emails and attachments — PDFs, Excel files, scanned documents — and nothing else
* A normalization agent that took that raw extraction and mapped it into a consistent schema
* An analysis agent that ran the normalized data against the bank's underwriting criteria
* A decision agent that proposed approve, deny, or flag for review — working only from the structured analysis

Each step was small enough to validate on its own. If extraction missed a field, we caught it before it corrupted the analysis. If a new attachment format came in, we updated one agent instead of rewriting a monolithic prompt.

*(If you're wondering what I use to bring pipelines like this to life, it's [Mastra](https://mastra.ai).)*

By the time we got to the part that actually required judgment — the underwriting decision — it was working on something that had already been shaped into a predictable form.

![The Pin Factory vs. A Multi-Agent System](/images/blog/pin-factory-multi-agent.png)

Once you start chaining steps like these together, the system stops behaving like a single pass and starts behaving like a pipeline. You can route based on outputs, reject results that don't meet a threshold, loop steps until they pass, or branch the process entirely. That only works if the underlying steps are already separated. Otherwise you're just layering more responsibility onto the same overloaded process.

---

## Knowing When to Split

The breakthrough with the underwriting agent came when we stopped thinking about it as a technology problem and started thinking about how the work already gets done inside a bank.

A human underwriting flow doesn't have one person doing everything. One person handles document collection — chasing down the emails, pulling attachments, making sure everything's there. Someone else extracts the data and gets it into a normalized form. An analyst runs the numbers against the bank's criteria. A reviewer checks the work. Finally, someone recommends approve or deny.

Nobody designed that process from a whiteboard. It evolved because banks learned, over decades, that one person trying to hold the whole thing in their head makes mistakes. The same mistakes we were seeing from our single agent.

So we modeled the agent system after the human one. An extraction agent that only worries about pulling data from documents. A normalization agent that only worries about getting that data into a consistent schema. An analysis agent that works from clean, structured inputs. A decision agent that proposes a recommendation based on the analysis.

Each agent got simpler. Each agent got more reliable. And when something went wrong, we knew exactly where to look.

---

## The Model Worth Carrying

The moment you find yourself writing a long prompt that tries to do multiple things at once — enforce structure, reason, format, handle edge cases — you're fighting the same battle Smith's lone pin-maker was fighting.

The real design skill in agent systems isn't making a powerful agent. It's noticing the moment a task stops being one job and becomes several. The smaller-step systems don't feel as impressive. They just keep working.
